CHAPTER 2 — THE SIMULATOR
BinMoments — a listening companion

(Written to be listened to, not studied. No diagrams, no code — just the idea.)


Let me start with what the simulator is for, because that frames everything else. You're building a system whose whole job is to notice when a sensor starts behaving abnormally. But here's the bind: to know whether your detector actually works, you need data where you already know the right answer — where you know, precisely, "an anomaly happened here, at this moment, of this size." Real sensor data never comes with that answer key. So you manufacture it. The simulator is the answer key generator. It produces fake-but-realistic temperature readings, and alongside them, a separate truth log saying exactly what was wrong and when. Later, you run your detector on the readings alone, and check whether it rediscovers what the truth log already knows. That comparison is the entire point.

Now, how does it build a realistic temperature signal? Think about real temperature over a year. There are two rhythms layered on top of each other. There's the slow one — the seasons — warm in summer, cool in winter, swinging gently across the whole year. And there's the fast one — day and night — warmest mid-afternoon, coldest before dawn, repeating every twenty-four hours. The simulator builds both as smooth waves. A slow wave for the seasonal swing, a fast wave for the daily swing, and it adds them together. So at any given moment, it can say "the underlying expected temperature right now is, say, twenty-two degrees" — because it knows where you are in the year and where you are in the day. Since this is a Brisbane-flavoured project, the seasonal wave is tuned for the southern hemisphere, so the peak lands in late January, not July.

But a pure sum of two smooth waves would be too perfect — real readings jitter. So on top of the smooth expected value, it sprinkles random noise, a little scatter around the line, the way a real thermometer wobbles reading to reading. And here's a detail worth appreciating: it keeps the smooth part and the random part separate internally. The clean wave is one thing; the noise is another thing added to it. That separation is not cosmetic — it's the key to everything that follows.

Here's why that separation matters, and this is the clever bit. When the simulator injects a fault — say, "shift the temperature up by five degrees for one day to mimic a drifting sensor" — it adds that five degrees on top of the exact same noise it would have generated anyway. It doesn't re-roll the dice. So if you run the simulation once clean, and once with the fault, the two runs are identical everywhere except inside that one day, and inside that day they differ by exactly five degrees, reading for reading. Nothing else moved. That means when your detector later flags that day, you can prove the flag is real — not a coincidence of different random noise, but a response to the one thing you changed. It's like having a perfectly controlled test signal where you've injected a known perturbation and everything else is held constant. That controllability is what makes the answer key trustworthy.

And there are a few different kinds of fault it can inject, because sensors fail in different ways. One is that shift I mentioned — the whole signal slides up or down, a drifting calibration. Another is the signal getting noisier — same average, but the scatter widens, like a sensor going flaky; the simulator does that by stretching the noise it already drew. A third is a stuck sensor — the reading freezes on one value and stops responding, which is a very real failure. And the fourth is a spike — a single wildly wrong reading, far outside anything normal, the kind of glitch that should make your system sit up. Each of these stresses a different part of your detector, which is why they're all there.

Every time it injects any of these, it writes a line in that separate truth log: what kind of fault, when it started, when it ended, how big. That log is the grading rubric. Your detector never sees it — it only sees the readings — but you hold it, and you grade against it.

Now, an important practical point: does the simulator create files? No, it does not. It's a generator that hands back two things in memory when you call it — a list of reading records, and the truth log. It doesn't write anything to disk on its own. The readings are produced in a structure that's ready to be written out as JSON — that's the format the rest of the system expects — but actually saving them to a file is a separate step. Each reading carries an instrument name, a timestamp for when the reading is about, a second timestamp for when the system would have received it, and the temperature value itself. When you run it to experiment, you get those records back in your program and can print them, eyeball them, or save them yourself — but nothing lands on disk automatically until you decide it should.

One last thing to set expectations. Because it's seeded with a fixed random starting point, running it twice with the same settings gives you exactly the same output, every time. That feels strange the first time — random data that's perfectly repeatable — but it's intentional, and it's the same principle as a reproducible simulation: same seed, same realization. It means your experiments are repeatable and your tests can check exact values. If you want different data, you change the seed; if you want the same data tomorrow, you don't touch it.

When you're ready to run it, the simplest thing is to call it for a short window — say one day, a few readings an hour — with one injected drift and one spike, and just print what comes back. You'll see the gentle daily wave, the noise wobble on top, the spike jumping out, and the truth log naming what you did.
